## What Went Well

- The product promise stayed focused: users can inspect value before signup.
- Public preview remained static, unauthenticated, and independent of Supabase configuration.
- The `/account` no-env behavior became a reusable safety pattern instead of a source of runtime failures.
- Account APIs consistently used server Supabase helpers, `getClaims()`, typed JSON, and `Cache-Control: no-store`.
- Code reviews caught meaningful issues early:
  - dependency/version and auth proxy scope in Story 1.1
  - privacy API caching and malformed JSON handling in Story 1.4
  - concurrent deletion request idempotency in Story 1.5
- Accessibility was pulled into the foundation before onboarding began, not postponed until after forms existed.

## Challenges

- The generated starter introduced defaults that needed cleanup before Routinelle-specific work could safely continue.
- Next.js 16 cache components required a specific dynamic rendering pattern: account work must remain behind `Suspense` and `connection()`.
- Missing local Supabase env vars limited live auth, consent, and deletion persistence verification.
- Browser-level viewport and accessibility automation is not configured, so Story 1.6 could only document the axe/viewport automation gap.
- Git tracking is not yet giving useful per-story diffs because the app and BMad artifacts are still untracked.

## Key Lessons

1. Treat generated starter code as scaffolding, not product code.
2. Keep public value preview separate from auth, Supabase, and account persistence.
3. For user-specific account APIs, default to authenticated server routes, typed error responses, and `no-store`.
4. Preserve controlled no-env behavior so local development and demos do not break the public experience.
5. Put accessibility rules in shared primitives early so later onboarding controls inherit them.
6. Story reviews should include edge cases for concurrency, malformed input, caching, and redirect safety.

## Technical Debt And Test Gaps

- Add browser automation for 320px/375px viewport checks before onboarding grows.
- Add axe or equivalent automated accessibility checks once a browser test stack exists.
- Decide whether to track the full app in git before more stories accumulate, so code review can rely on clean diffs.
- Live Supabase auth/persistence behavior still needs validation with real local or staging env vars.
- The current account data deletion flow records deletion requests; future profile/routine storage stories must honor that state.
